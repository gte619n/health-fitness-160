# IMPL-03: Firestore persistence layer

> **2026-05-22 revision**: Testcontainers-based integration tests are
> removed. The persistence module's Firestore-backed tests proved
> awkward to run on developer machines that don't have a Docker daemon
> in a standard socket location, and the deployed environment turned
> out to be the more efficient place to validate real Firestore
> behavior. Local development now hits Cloud Firestore directly (the
> `(default)` database); the deployed Cloud Run service points at a
> separate `production` database via `FIRESTORE_DATABASE_ID=production`
> in `backend/cloudbuild.yaml`. All other sections of this spec
> describe the original landing — the file is preserved as a record of
> what was planned and what changed.

## Goal

Replace the in-memory `UserRepository` shim from IMPL-02 with a Firestore-
backed implementation, wire the Firestore client into the backend, and lock
in the document layout the rest of the persistence layer will follow.
Also land empty domain records and repository scaffolds for the four
health-data entities IMPL-04+ will populate — `Workout`, `DailyMetric`,
`Location`, `Equipment` — so the collection paths and mapping conventions
are decided once.

Locations and Equipment exist so workout planning can be constrained to
the gear actually available where the user is training: a "home gym" has
adjustable dumbbells up to 50 lb, the "hotel" has bands, the "office gym"
has a full rack and plates. Workouts get a nullable `locationId` so this
context can be attached when known.

Users provisioned by the existing `UserProvisioningFilter` from IMPL-02
land as real Firestore documents at `users/{sub}` from this IMPL forward.

## Scope

In scope:

- A Firestore-backed `UserRepository` impl in the `persistence` module
  replaces `InMemoryUserRepository`.
- Firestore client bean (`com.google.cloud.firestore.Firestore`) configured
  from Application Default Credentials in production and from
  `FIRESTORE_EMULATOR_HOST` locally.
- Document-mapper conventions: `Instant` ⇄ `com.google.cloud.Timestamp`,
  `LocalDate` ⇄ ISO-8601 string used as document ID for date-keyed
  collections, audit fields written as `FieldValue.serverTimestamp()` on
  every write.
- Four new domain records in `core` — `Workout`, `DailyMetric`,
  `Location`, `Equipment` — plus their matching repository interfaces.
- Persistence-side impls for all five repositories (`User`, `Workout`,
  `DailyMetric`, `Location`, `Equipment`). The `User` impl is fully
  functional; the other four are scaffolds that throw
  `UnsupportedOperationException` until the ingestion / write paths land
  in later IMPLs. The scaffolds exist now so the document layout, naming,
  and wiring decisions are made once.
- Integration-test setup using the Firestore emulator via Testcontainers.
  Existing unit tests that need a repository continue to use a small
  in-memory fake (`InMemoryUserRepository` is repurposed as a test fake,
  not a Spring `@Repository`).
- Remove `spring-cloud-gcp-starter-data-firestore` from
  `gradle/libs.versions.toml` (it was declared in IMPL-00 but is unused;
  see [ADR-0003](../decisions/ADR-0003-firestore-client-choice.md)).

Out of scope (deferred):

- Webhook ingestion (`integrations/webhook`) — IMPL-05.
- Google Health API client (`integrations/googlehealth`) — IMPL-04.
- Read-side controllers for workouts/metrics — separate IMPL once
  ingestion produces data.
- Firestore composite indexes — none required for the writes in this IMPL.
  When IMPL-04 adds time-range queries, the required indexes ship with
  that IMPL.
- Per-collection backup or TTL policies.

## Decisions

| Topic | Decision |
|---|---|
| Firestore client | `com.google.cloud:google-cloud-firestore` (direct, synchronous). See [ADR-0003](../decisions/ADR-0003-firestore-client-choice.md). |
| Reactive Firestore | Out. `spring-cloud-gcp-starter-data-firestore` removed from `libs.versions.toml`. |
| Document layout | `users/{sub}`, `users/{sub}/workouts/{workoutId}`, `users/{sub}/dailyMetrics/{yyyy-MM-dd}`, `users/{sub}/locations/{locationId}`, and **top-level** `equipment/{equipmentId}`. Equipment is its own root collection because it's reusable across users and locations — see [Equipment design](#equipment-design). |
| Document IDs | `User.userId` = Google `sub` claim. `Workout.workoutId`, `Location.locationId`, `Equipment.equipmentId` = UUIDs generated by the writer (when ingestion adds real sources, e.g. Google Health record IDs, the writer chooses). `DailyMetric` document ID = ISO date (`yyyy-MM-dd`). |
| Audit fields | `createdAt`, `updatedAt` are Firestore `Timestamp` columns written as `FieldValue.serverTimestamp()`. Java reads return `Instant`. |
| `User` record shape | Unchanged from IMPL-02: `(userId, email, displayName, createdAt, updatedAt)`. Mapper writes `email` and `displayName` as strings. |
| Equipment specs | `Equipment.specs` is `Map<String, Object>` — flexible per-category bag (min/max weight, increment, plate set, resistance levels). Schema crystallizes when planning logic actually consumes it. No sealed-interface-per-category yet. |
| Equipment ownership | `Equipment.ownerId` is nullable. `null` = system-wide catalog entry (any user can reference it). Set to a userId = custom equipment that user defined. Lets the catalog grow without schema changes. |
| Location ↔ Equipment | `Location.equipmentIds` is a `List<String>` of references into the top-level `equipment` collection. Bounded (10–30 typical). `where equipmentIds array-contains <id>` query works if we ever need "who else has this barbell". |
| Workout ↔ Location | `Workout.locationId` is nullable. Set when the user logs from the app with a location selected, null when Google Health ingestion has no concept of place. |
| Repository return types | `Optional<T>` for single-doc reads, `List<T>` for collection reads. No Reactor types in `core` or `persistence`. |
| Naming | Persistence impls do **not** carry a `Firestore` prefix — we will only ever use Firestore. Interface and impl share the same simple name (`UserRepository`) in different packages (`core.user`, `persistence.user`); the impl uses the fully qualified interface name in its `implements` clause. |
| Scaffolds | The Workout, DailyMetric, Location, and Equipment repository impls throw `UnsupportedOperationException("writes land with a later IMPL")`. Records and interfaces compile fully; downstream code can reference them now. |
| Production credentials | Application Default Credentials. Cloud Run runtime SA already has `roles/datastore.user`. No service-account JSON files in the repo. |
| Local credentials | Firestore emulator via `FIRESTORE_EMULATOR_HOST=localhost:8080` (set in `dev.sh`). Real Firestore in `us-central1` reachable only via deployed env. |
| Test credentials | Testcontainers boots `gcr.io/google.com/cloudsdktool/cloud-sdk:emulators` and sets `FIRESTORE_EMULATOR_HOST` per JUnit suite. |
| Transactions | Not used in this IMPL. `provisionIfAbsent` keeps its current read-then-write shape; the race is benign because the document body is deterministic (same `sub` → same fields). Revisit if/when a workout-summary aggregate lands. |

## Equipment design

Equipment lives in its **own top-level collection** at `equipment/{equipmentId}`,
not as a subcollection under a location. Three reasons:

1. **Reuse.** A "PowerBlock 5-50lb dumbbell set" has the same specs no
   matter who owns one. Defining the spec once and referencing it from
   any location avoids re-typing the same equipment on every user's home
   gym.
2. **System catalog.** Over time we expect a curated catalog of common
   equipment (Concept2 RowErg, standard Olympic barbell, etc.) that users
   pick from rather than redefine. Top-level placement lets these
   coexist with user-defined entries via `ownerId == null`.
3. **Equipment-driven planning.** When workout planning logic eventually
   wants "which exercises can I do at my hotel today?", it needs to read
   equipment specs across many locations and users with the same query
   shape. A top-level collection makes that a single index.

Locations hold an `equipmentIds: List<String>` array of references.
Reading "all equipment at this location" is a one-doc read of the
location plus a batched `getAll` on the referenced equipment docs. With
typical location sizes (≤30 items) this is well inside Firestore's
performance envelope.

Equipment is not yet *written* in this IMPL — only the model, the
collection path, and the repository surface are locked in. Population
flows (admin seeding the system catalog, user creating a custom piece,
attaching equipment to a location) land in a later IMPL once the
location-management UI exists. Until then, repository methods throw
`UnsupportedOperationException`.

  ```java
  public record Equipment(
      String equipmentId,
      String ownerId,              // nullable; null = system catalog, set = user-owned custom entry
      String name,                 // "PowerBlock dumbbells 5-50lb", "Concept2 RowErg PM5"
      String category,             // "dumbbell" | "barbell" | "machine" | "cardio" | "bands" | "bodyweight" | "kettlebell" | "rack" | "bench" | "other"

      String notes,                // free text

      Instant updatedAt
  ) {}
  ```

  shapes like:

  - barbell: `{ "barWeightLb": 45, "plateWeightsLb": [2.5, 5, 10, 25, 35, 45] }`
  - cardio: `{ "type": "rower", "hasHeartRateSensor": true }`
  - bands: `{ "resistanceLevelsLb": [10, 25, 50] }`

  Crystallizing into typed sub-records is deferred until the planning
  logic that reads these specs actually exists.

**`persistence` module**:

Naming rule: persistence classes do **not** carry a `Firestore` prefix.
The Spring config and the type-conversion utility do, because they
literally describe Firestore wiring — but repositories use the same
simple name as their interface, distinguished by package.

- `persistence/src/main/java/com/gte619n/healthfitness/persistence/FirestoreConfig.java`
  `@Configuration` class. Single bean:
  ```java
  @Bean
  Firestore firestore(@Value("${app.gcp.project-id}") String projectId) {
      return FirestoreOptions.getDefaultInstance().toBuilder()
          .setProjectId(projectId)
          .build()
          .getService();
  }
  ```
  The Firestore client automatically honors `FIRESTORE_EMULATOR_HOST` when
  set — no separate code path for emulator vs production.
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/FirestoreMapper.java`
  Package-private static helpers:
  - `Instant toInstant(Object timestampValue)` — accepts
    `com.google.cloud.Timestamp` and returns `Instant`. Returns `null` on
    `null`.
  - `Object serverTimestamp()` — returns `FieldValue.serverTimestamp()`.
  - `String toDocumentId(LocalDate date)` — `date.toString()`
    (ISO-8601 `yyyy-MM-dd`).
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/user/UserRepository.java`
  Replaces `InMemoryUserRepository`:
  ```java
  package com.gte619n.healthfitness.persistence.user;

  @org.springframework.stereotype.Repository
  public class UserRepository implements com.gte619n.healthfitness.core.user.UserRepository {
      private static final String COLLECTION = "users";
      private final Firestore firestore;
      // constructor injection
      // findById -> firestore.collection(COLLECTION).document(userId).get().get()
      // save     -> firestore.collection(COLLECTION).document(user.userId()).set(toMap(user)).get()
      // toMap writes: email, displayName, createdAt (serverTimestamp on first
      // save only), updatedAt (serverTimestamp every save)
  }
  ```
  Blocking on `ApiFuture#get()` is intentional and matches the
  virtual-threads model — IO suspends a virtual thread, not a carrier
  thread.
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/workout/WorkoutRepository.java`
  Implements `com.gte619n.healthfitness.core.workout.WorkoutRepository`.
  All methods throw `UnsupportedOperationException("workout writes land
  with a later IMPL")`. Bean still registered so the module wires up
  cleanly.
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/metric/DailyMetricRepository.java`
  Same pattern — throws until ingestion lands.
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/location/LocationRepository.java`
  Same pattern — throws until the location-management UI / API lands.
  Reads/writes against `users/{userId}/locations/{locationId}`.
- `persistence/src/main/java/com/gte619n/healthfitness/persistence/equipment/EquipmentRepository.java`
  Same pattern — throws until equipment management lands. Reads/writes
  against the **top-level** `equipment/{equipmentId}` collection — not
  user-scoped.
- **Delete** `persistence/src/main/java/com/gte619n/healthfitness/persistence/user/InMemoryUserRepository.java`
  from the production source set. Move an identical fake to the **test**
  source set (`persistence/src/test/java/.../InMemoryUserRepositoryFake.java`)
  for unit tests that don't need a full Firestore. The fake retains its
  `InMemory` prefix because *it is* a different impl strategy, used only
  in tests.

### Tests

- `UserRepositoryIntegrationTest` (Testcontainers emulator):
  - `save` then `findById` returns the same `User` with `createdAt` and
    `updatedAt` populated.
  - Repeating `save` for the same `userId` is idempotent and bumps only
    `updatedAt`.
  - `findById` for an unknown user returns `Optional.empty()`.
- `UserProvisioningFilterIntegrationTest`:
  - With the existing JWT auth path plus the emulator-backed repository,
    a single authenticated request to `/api/me` produces exactly one
    `users/{sub}` document in Firestore. Repeating the request does not
    create a second document.
- `FirestoreMapperTest` (unit, no emulator):
  - `toInstant` handles `null`, `Timestamp`, and unexpected types
    (throws `IllegalArgumentException` for the third).
  - `toDocumentId(LocalDate)` produces `yyyy-MM-dd`.
- `ScaffoldRepositoryTest` (one tiny test per stub):
  - Asserts every `Workout`, `DailyMetric`, `Location`, `Equipment`
    repository method throws `UnsupportedOperationException`. Acts as
    documentation that these are deliberate stubs, and will start failing
    the day someone deletes a `throw` line — at which point the test
    becomes the place to add real coverage.
- Existing `SecurityConfigTest`, `DevHeaderAuthTest`, and
  `WhoAmIControllerTest` continue to pass against the in-memory fake
  pulled in via `@TestConfiguration`.

The emulator container is started once per test class
(`@Testcontainers @Container static`) and reset between tests by deleting
the `users` collection — Firestore emulator is fast enough that the full
test suite stays under ~10s on a developer laptop.

## Open questions resolved before implementation

- **Sync vs reactive Firestore** — Sync. ADR-0003.
- **Health data scope** — Stubs only (records + repository interfaces
  + throwing impls). Behavior lands with IMPL-04.
- **Server timestamps vs `Instant.now()`** — Server timestamps. Removes
  clock-drift between containers and matches how the eventual Google
  Health ingestion will write records.
- **Collection layout** — User-scoped subcollections for User-owned data
  (`users/{sub}/...`); top-level for shared catalog data (`equipment/...`).
- **Equipment scope** — Top-level `equipment/{equipmentId}` collection,
  not nested under user or location. Reusable across users and locations.
  `Location.equipmentIds` is a `List<String>` of references; reading
  "all equipment at this location" is one doc read plus a batched
  `getAll`. See [Equipment design](#equipment-design).
- **System vs user-owned equipment** — Single collection,
  `Equipment.ownerId` is nullable. `null` = system catalog, set = custom.
  Avoids a second collection until the access patterns demand one.
- **Equipment specs as typed sub-records vs untyped `Map`** — Untyped
  `Map<String, Object>` for now. Examples documented per category; typed
  records added when planning logic actually reads them and the schema
  pressure forces the call.
- **`FirestoreUserRepository` vs `UserRepository` for the impl class** —
  Same name as the interface, different package. We will only ever use
  Firestore for persistence; the technology prefix would be noise. The
  one syntactic cost is the FQCN in the `implements` clause of each
  impl, paid once per file.
